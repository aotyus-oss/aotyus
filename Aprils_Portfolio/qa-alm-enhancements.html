Chapter 01

Where QA
started.

When I took over as QA Manager at Billtrust, there was no structure at all. There were no test cases, no standards for how issues were reported, and no organization in the SRD for QA activities. Testing was entirely ad hoc — testers verified things informally with no documented scope, no defined format, and no record of what had actually been tested. Comments were inconsistent. No one knew what anyone else had covered.

The first phase of improvement was building structure into the SRD itself. I led the QA team to start writing test cases, follow a standard format for how test cases were written, and adopt a consistent approach to reporting issues inside the spreadsheet. Over time, the SRD evolved from an unorganized document into a structured QA record — with defined rows for test cases, consistent issue reporting, and a shared understanding of how QA activities were tracked.

But even a well-structured spreadsheet has limits. As the team grew and projects became more complex, the SRD's structural constraints became impossible to work around. Everything still lived in one file. Nothing was connected. Nothing was reportable.

The problem with the SRD
A spreadsheet is a document, not a system. It has no lifecycle, no audit trail, and no reporting mechanism. As projects grew in complexity, the SRD became harder to maintain, harder to track, and impossible to query. QA needed a real platform.
📄
Requirements in a spreadsheet
Every requirement is a row in the SRD with an Ownership/Status column to indicate who owns it. There is no Salesforce record, no lifecycle, and no way to track a requirement's status or history outside of the spreadsheet.
✍️
Test cases written by hand
Test cases were written manually from the SRD requirements. There was no automated generation.
📋
Results recorded in the SRD
Test results were recorded directly in the SRD alongside the requirements, test cases, execution results, and issues. The SRD had structure, but all reporting was limited to what could be done inside a spreadsheet.
🔍
Traceability depended on manual discipline
Traceability existed but depended on everyone manually entering the correct requirement ID, test case ID, and execution result in the right place.
📊
Reporting required Excel knowledge
Reporting required knowledge of Excel formulas and pivot tables. Results were only as accurate as the data entered. If statuses were not updated correctly, the reporting reflected that.
🔗
Everything lived on one sheet
Everything lived on one sheet. Requirements, test cases, results, and issues were all connected but as projects grew, managing a single large spreadsheet became increasingly difficult to maintain and navigate.
Chapter 02

Where it
is now.

The QA ALM system replaced the SRD with a structured, connected platform built on Salesforce. Test cases, test plans, test runs, test case executions, and issues are now Salesforce records. QA data is reportable, traceable, and connected end to end.

AI is already handling the two most time-consuming manual tasks: generating test cases from requirements and analyzing requirements for gaps and ambiguities before testing begins. Both are live and in use today.

What was built
The QA ALM system introduced a structured data model inside Salesforce: Sub-case → Test Plan → Test Case → Test Case Execution → Issue. Every record is connected. A test case links to a requirement. An execution links to a test case. An issue links to an execution. For the first time, QA data could be reported on, filtered, and traced from requirement to result.
🏗️
Structured Salesforce data model
Test cases, executions, issues, and results are all Salesforce records with defined fields, relationships, and lifecycle stages.
🔗
End-to-end traceability
Every test case links to a requirement. Every execution links to a test case. Every issue links to an execution. The full chain is visible inside one platform.
📊
Salesforce reporting
Pass rates, blocked counts, issue counts by type, and test cycle progress are all available as Salesforce reports.
🤖
AI test case generation
Claude reads the SRD mapping tab and generates CAPTURE and DISPLAY test cases based on documented rules. The QA Analyst reviews and imports.
🔎
AI requirement analysis
Claude reads the SRD and identifies gaps, ambiguities, and clarifying questions before any test case is written.
📂
AI data analysis
Claude reads the SRD and client data files, identifies missing data across all mapped positions, and identifies negative values.
What is still missing
Requirements still live in the SRD. Every test case references a Requirement ID that points back to a row in a spreadsheet outside of Salesforce. The system is structured and connected, but it is not yet complete.
Chapter 03

Where it
is going.

The next phase closes the remaining gap and extends AI deeper into every stage of the QA workflow. The Requirement Object — which brings requirements into Salesforce as structured records — is currently in the planning stage. An MCP server connection that allows Claude to read from and write to Salesforce directly is currently in progress.

Together, they will make it possible for Claude to read requirements from Salesforce, generate test cases, and execute test cases — with no file exports, no import scripts, and no manual steps. Everything will stay inside one system.

Next. New Salesforce Object
The Requirement Object
A Salesforce object that gives every requirement its own record, owner, status, and lifecycle; created directly inside the Test Plan.
Planned
Requirements are currently text fields, a Requirement ID pointing back to a row in the SRD. Business Analysts create them inside the Test Plan. They link to test cases, executions, and issues. They have owners, statuses, and audit history. The SRD is no longer the system of record for what is being tested.
📋
Created inside the Test Plan
Business Analysts create Requirement records inside the Test Plan.
🏷️
Owner, status, lifecycle
Each requirement has an assigned owner, a status field, and full edit and audit history inside Salesforce.
🔗
Linked to test cases and executions
Test cases link directly to the Requirement record. Executions trace back to the requirement.
📈
Reportable inside Salesforce
Once requirements are Salesforce records, they can be reported on like any other object. Which requirements have test coverage, which have open issues, and which are blocked are all visible in Salesforce reports.
🔓
Removes the SRD dependency
Requirements become self-contained inside Salesforce. The SRD is no longer required to understand what is being tested or to trace results back to requirements.
🤖
A prerequisite for the MCP vision
For Claude to read requirements from Salesforce via MCP, requirements must exist as Salesforce records. The Requirement object is what makes that possible.

Updated data model with the Requirement object

The Requirement object sits between the Test Plan and the Test Case, linking test coverage back to the source requirement that defined what to build.

Sub-case Custom Object
The parent container for a project. Each Sub-case has exactly one Test Plan and can have many Issues reported outside of the testing process entirely.
Test Plan Custom Object
One per Sub-case. Contains many Test Cases, many Test Runs, many Test Case Executions, and many Issues reported during testing.
Requirement New Object
Created directly inside the Test Plan. Each requirement has its own record, owner, status, and full audit trail. Replaces the freetext Requirement ID field.
Test Case Custom Object
The reusable test scenario. Linked to one or more Requirement records.
Test Case Execution Custom Object
Result of running a test case in a Test Run. Traces back to the Requirement via the Test Case link, enabling requirement-level pass/fail reporting.

The MCP vision. Claude reads and writes Salesforce directly

When the Requirement object exists and an MCP server connection is established, the test case generation workflow collapses from six steps to five and every manual handoff step disappears.

Today - six steps, three manual handoffs:

01
Requirements authored in SRD
Business Analyst writes requirements in a spreadsheet outside of Salesforce.
02
QA Analyst uploads SRD to Claude
Claude reads the file and generates test cases using the generation rules.
03
Excel file exported and reviewed
QA Analyst downloads AI_Generated_TestCases.xlsx and completes the human review checklist.
04
File moved to Ready for Import folder
QA Analyst manually moves the reviewed file to trigger the Power Automate flow.
05
JAR tool imports test cases into Salesforce
Power Automate executes the BAT script, which calls the JAR and creates the test case records in Salesforce.
06
QA Analyst executes test cases
QA Analyst manually executes each test case in Salesforce, recording pass, fail, or blocked results and logging any issues found during execution.

With MCP - five steps, one human checkpoint:

01
Requirements authored in Salesforce Requirement Object
Business Analyst creates Requirement records directly inside the Test Plan in Salesforce.
02
Claude reads requirements from Salesforce MCP Server
Claude connects to Salesforce via MCP and reads the Requirement records directly. No file upload required.
03
Claude generates test cases and presents for review
Claude applies the generation rules and presents the test cases for the QA Analyst to review and confirm before anything is written to Salesforce.
04
Claude writes test cases directly into Salesforce MCP Server
Once confirmed, Claude creates the test case records in Salesforce, links them to the test plan, and adds them to the test run for execution.
05
Claude executes test cases and auto-drafts issues on failure MCP Server
Claude executes each test case in Salesforce, records pass, fail, or blocked results, and automatically drafts an Issue record for any failed execution. The QA Analyst reviews and submits each drafted issue.
What this requires
This vision depends on two things coming together: the Requirement object being built so requirements are structured Salesforce records, and an MCP server connection being established between Claude and Salesforce.

AI across the full QA workflow

Beyond the MCP connection, AI is being extended into every remaining manual stage of the QA process - execution, issue management, and reporting.

Execution
Automated test execution with pass/fail recording
In Progress
AI executes test cases and records the execution results directly on the Test Case Execution record. The QA Analyst reviews AI-recorded results rather than performing every execution manually. For complex scenarios, the QA Analyst retains execution control.
Eliminates the most time-consuming manual step in the QA cycle
Issue Management
Auto-draft issue records on failed executions
Planned
When a test case execution is marked as failed, the system detects the failure and drafts an Issue record with the title, description, Issue Type, and Category. The QA Analyst reviews and submits rather than building the issue from scratch.
Reduces time from failure detection to issue submission
Reporting
Natural language reporting
Planned
QA Analysts and managers ask questions in plain English. "How many issues were deferred in Post UAT2?" or "Which test cases had the most failures this quarter?", and AI queries the Salesforce data and returns a direct answer. No report builder required. Makes QA data accessible to anyone on the team, not just people who know how to configure Salesforce reports.
Makes QA data accessible to the whole team without Salesforce report expertise
Reporting
AI release readiness reporting
Planned
AI reviews the full state of the Test Plan — pass rates, open issues, blocked executions, and deferred items — and generates a structured release readiness report that helps the QA Analyst make the go/no-go decision. No manual data gathering required.
Speeds up the release sign-off process by giving the QA Analyst a complete picture in seconds
