The Problem

QA lived inside
a spreadsheet.

Before this system was built, QA was managed entirely within a shared Excel spreadsheet called the SRD — Standard Requirements Document. Every project had its own SRD, and the same document was reused across every project cycle including post go-lives.

The SRD was structured with requirements in their own rows. Underneath each requirement, the same rows served multiple purposes simultaneously — test cases, test case executions, execution results, and issues all lived together in those rows, distinguished only by cell values and who the row was assigned to.

Core limitation
A single spreadsheet row had to function as a test case, an execution record, a result, and an issue log all at once. There was no separation between these concepts — everything was inferred from what was written in the cells.

01
No structured issue tracking
Issues were free-text comments in a cell — not categorized, not typed, not tracked as distinct records. There was no way to report on issue trends or patterns.
02
No notifications
When a row was assigned to an engineer or BA, there was no automated alert. Engineers and QA Analysts had to manually monitor the spreadsheet to know when action was needed.
03
No audit trail
Comments could be overwritten and history was lost. There was no reliable record of who changed what, when, or why — making it difficult to reconstruct decisions after the fact.
04
No test case reuse
Test cases were rewritten or manually copied for each project cycle. There was no mechanism to reuse existing test cases across projects or post go-live cycles.
05
No reporting or dashboards
To understand the status of a release, someone had to read through the spreadsheet row by row. There was no way to get a high-level view of test coverage, pass rates, or open issues.
06
Manual test case IDs
Each test case required a manually assigned ID in the format TC-reqid-ffdid-xx. This introduced risk of duplicates, formatting errors, and gaps in the sequence.
07
One column forced to do two jobs
The Ownership/Status column had to simultaneously communicate who owned a row and what the execution status was. There was no timestamp and no history of who changed it or when.
My Role

Framework designer.
Not the builder.

My role on this project was to design the QA framework and author the functional requirements that guided Salesforce implementation. I identified the limitations of the SRD process, defined what a replacement system needed to do, and directed SalesOps Engineering on how it should work.

What I Built

A proper data model
for every QA concept.

The core design principle was simple: give every concept its own identity. Test cases, execution runs, results, and issues are no longer crammed into the same spreadsheet rows — each is a distinct Salesforce record with its own fields, relationships, and trackable status.

Sub-case Salesforce object
The parent container for a project. Each Sub-case has exactly one Test Plan and can have many Issues reported outside of the testing process entirely.
Test Plan Custom object
One per Sub-case. Contains many Test Cases, many Test Runs, many Test Case Executions, and many Issues reported during testing.
Test Case Custom object
Belongs to a Test Plan directly. Can also be reused across many Test Plans via the Test Case Plan junction. Test Case IDs are automatically generated by Salesforce — no manual assignment required.
Test Case Plan Junction object
Links Test Cases to Test Plans in a many-to-many relationship. Enables a single Test Case to be reused across multiple Test Plans without duplicating the record.
Test Run Custom object
Represents a single execution cycle within a Test Plan. Contains many Test Case Executions — one per test case being run in that cycle.
Test Case Execution Custom object
Records the execution of a specific Test Case within a Test Run. Captures pass/fail/blocked status and can have many Issues logged against it.

Issue object — full field design

Issues can be logged at three levels — Sub-case, Test Plan, and Test Case Execution — depending on where and when they are found. The Issue object was designed with six sections covering the full lifecycle of an issue from discovery through resolution.

Issue Summary
Issue ID
Auto-generated unique identifier assigned by Salesforce on record creation.
Issue Status Required
The current status of the issue. Drives visibility and workflow across the team.
Issue Lifecycle Status
Indicates the current working state of the issue within its lifecycle. Picklist: Pending Dev | In Development | Researching | Pending QA | Validating | Pending BA Review | Reviewing | Needs Feedback | Deferred | Completed.
Issue Type Required
Classifies the nature of the issue. Picklist: Requirement · Defect · Question/Clarification · Bad Data · Missing Data.
Issue Category
A filtered sub-classification based on Issue Type. Requirement issues map to: New Requirement | Updated Requirement | Unclear Requirement | Missing Requirement | Incomplete Requirement | Conflicting Requirement. All other types map to: Data Mapping | Reports | Document Display | Email Template | Routing.

Design decision
Requirement issues are separated from technical issues intentionally. This means the system captures both system defects and requirement quality problems — giving teams visibility into gaps that traditional defect-only trackers would miss entirely. Because Found In and Fixed In share the same project phase picklist, you can compare the two fields on any issue to see exactly how many cycles passed between when a problem was discovered and when it was fixed.
How It Works

The QA workflow,
end to end.

Each QA Analyst owns their project end to end — from writing test cases through to release sign-off. The system supports the full project cycle including post go-live testing.

01
Sub-case is created
A Salesforce Sub-case is opened for the project. This is the top-level container that everything else lives under. One QA Analyst is assigned to the Sub-case for the duration of the project.
02
Test Plan is created
A Test Plan is created under the Sub-case. Each Sub-case has exactly one Test Plan. This is where the scope of testing for the project is defined.
03
Test Cases are added to the Test Plan
Test Cases are added to the Test Plan via the Add Related Test Cases button. The QA Analyst has three options for how to create or add a test case.
Option A — Select an existing Test Case from the system and add it to the plan (reuse)
Option B — Create a new Test Case directly within the Test Plan
Option C — Create a Test Case independently outside the plan, then add it later
04
Test Runs are created and test cases added
Test Runs are created under the Test Plan. Test cases are added to the test runs from the Related Test Cases list. A Test Case Execution record is created for each test case that is added to the test run. The QA Analyst executes each Test Case Execution record. Each Test Case Execution record is marked as passed, failed, or blocked.
During ad hoc testing, a QA Analyst can create a new Test Case directly from a Test Case Execution — when a scenario arises that wasn't planned for. The new Test Case is then manually added to the Test Plan for proper tracking.
05
Issues are logged
If an issue is found during execution, it is logged directly against the Test Case Execution with a type and category. Issues can also be logged at the Test Plan level (during testing but outside a specific execution) or at the Sub-case level (found completely outside of testing).
Issues are assigned to either an engineer or a BA — the tester makes this judgment call based on the nature of the issue. Assignment triggers both an email notification and a Chatter notification to the assignee.
06
Project Cycle Release sign-off
The QA Analyst is responsible for making the Project Cycle Release decision, ensuring all quality standards have been met. A Project Cycle is approved for release when the following criteria are satisfied:
  • All planned test cases have been executed
  • All requirements validated and approved
  • No critical defects remain open
  • Medium defects have been reviewed and either accepted or deferred with approval
  • Regression testing has been completed successfully with no outstanding issues
