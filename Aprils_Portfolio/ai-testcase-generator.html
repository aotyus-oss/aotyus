The Problem

Mapping testing was
entirely ad hoc.

Before the AI Test Case Generator, mapping testing was done entirely ad hoc. There were no mapping test cases and testers verified fields informally with no documented structure, no defined scope, and no record of what was actually tested. SRD mapping tabs can contain many rows. Each requiring a test case that references the correct position, BT Property, FFDID, and record type context. Without written test cases, there was no way to know what another tester had covered, no way to reproduce a test, and no audit trail linking results back to specific mapping requirements.

Core limitation
Mapping testing had no structure at all. There were no written test cases, no defined coverage, and no traceability. If someone tested a field, there was no record of it. If a defect surfaced later, there was no way to determine whether that mapping had ever been tested or by whom.
01
No test cases, ever
Mapping testing was never documented. There were no test cases written for any mapping field. Testers verified values informally with no defined scope, no structure, and no record of what was checked.
02
No visibility into what others tested
Because nothing was written down, there was no way to know what a previous tester had covered. Coverage was completely unknown. There was no way to confirm what had been verified and what had been missed.
03
Nothing to go back to
If a mapping defect surfaced during UAT or after go-live, there was nothing to reference. No test cases, no results, no record of what was checked, just the defect itself.
The Solution

Structured mapping test coverage
for the first time.

The AI Test Case Generator introduced formal mapping test coverage where none had existed before. Instead of ad hoc verification with no documentation, every eligible row in the SRD mapping tab now produces a written, structured test case linked to the correct FFDID, record type, position, and BT property, ready to execute and trace back to the requirement.

The skill reads the mapping tab, evaluates every row against a complete set of generation rules, and exports the results to a downloadable Excel file alongside a full execution trace and skip log, giving the team documented coverage for the first time in the process.

Important distinction
The skill applies rules. It does not interpret requirements. Every test case it generates follows an exact sentence format derived from the SRD data. A human reviewer checks the output before anything reaches Salesforce. The skill handles generation and consistency. The QA Analyst reviews the output, confirms the test cases are accurate and executable, and approves the file before it is imported into Salesforce.

Generation Rules

Every row evaluated
against the same rule set.

The skill applies a complete, documented rule set to every row in the mapping tab. Rules cover row eligibility, record type tracking, BT property parsing, and test case generation, applied in the same order, every time, with no variation between QA Analysts or projects.

Row Skip Rules
Ineligible rows are excluded before generation
Rows are skipped when Ownership/Status is INFORMATIONAL, N/A, (Select One), or blank. Any combination of Appears On Bill and Needs To Be Captured that doesn't meet capture criteria is also excluded, with a reason code written to the skip log.
Record Type Tracking
Record type applies to every row below it until a new one is found
When Columns A through D are blank and Columns E through M form a merged cell, that text becomes the active record type. It is used in every test case sentence generated below it until the skill finds the next record type row in the tab, which then takes over.
BT Properties Parsing
One test case per property, regardless of delimiter
A BT Properties cell may contain multiple values split by commas, semicolons, pipes, slashes, newlines, or tabs. Each is trimmed and treated as an independent property. Depending on the Appears On Bill and Needs To Be Captured values, each property generates its own CAPTURE test case, or both a CAPTURE and a DISPLAY test case. No values are merged or skipped.
Case 1
Appears On Bill = Y and Needs To Be Captured = Y
Generates both a CAPTURE test case and a DISPLAY test case. If BT Properties contains multiple values, one CAPTURE test case and one DISPLAY test case are generated per property.
Cases 2 · 3 · 4
Appears On Bill is not Y, Needs To Be Captured = Y
When Needs To Be Captured = Y but Appears On Bill is N, blank, or N/A, only a CAPTURE test case is generated per BT property. DISPLAY is not generated because the value does not appear on the bill. This covers three scenarios: Case 2 (Appears On Bill = N), Case 3 (Appears On Bill = blank), and Case 4 (Appears On Bill = N/A).
Blank Normalization
Cell values are cleaned up before any rules are applied
Some SRD cells may look empty but contain whitespace-only values, the string 'nan', actual blanks, or a Python None value instead of being truly blank. Before applying any skip logic or case rules, the skill standardizes all of these variations to a true empty value so that rows are never incorrectly skipped or processed due to how the SRD was formatted.
Test Case Format

Two test case types.
One consistent sentence format.

Every test case the skill generates follows a prescribed sentence structure scoped to an FFDID and template type, anchored to a record type and position, and tied to a named field or BT property. The format never varies, regardless of which row or tab it came from.

CAPTURE Needs To Be Captured = Y
When generated
Needs To Be Captured = Y for any value of Appears On Bill (Cases 1, 2, 3, 4)
Test Case Title format
Mapping for <Field Description> - Doc Prop <BT Property>
Test Case sentence
For FFDID [FFDID] and for a [TEMPLATE TYPE] record, verify that the value passed in [RECORD TYPE] Position [POSITION] is captured and stored in bt property [BT_PROP].
DISPLAY Appears On Bill = Y
When generated
Appears On Bill = Y AND Needs To Be Captured = Y (Case 1 only)
Test Case Title format
Mapping for <Field Description> (DISPLAY)
Test Case sentence
For FFDID [FFDID] and for a [TEMPLATE TYPE] record, verify that the value passed in [RECORD TYPE] Position [POSITION] is displayed on the template in Field [FIELD_NO] (Field Name [FIELD_NAME]).
Output file structure
The Excel output creates one sheet per FFDID named Mapping TC FFDID [number]. Within each sheet, test cases are grouped by record type row. Column A = Req ID, Column B = Test Case Title, Column C = Test Details. No duplicate test cases are written. The file is named AI_Generated_TestCases_YYYYMMDD_HHMMSS_AM/PM.xlsx using the actual generation timestamp.
Human Review

AI generates.
Humans approve.

Before a single test case reaches Salesforce, a QA Analyst reviews the generated Excel file against a 5-point checklist. Nothing moves forward without human sign-off. The skill processes every row in the mapping tab and produces test cases in the same format every time. The QA Analyst reviews the output, confirms the test cases are accurate and executable, and approves the file before it is imported into Salesforce.

Human-in-the-Loop Checklist
✅
Requirement coverage complete
Every eligible SRD row has a corresponding test case in the output file. No rows that should have generated a test case are missing.
✅
Test case format correct
All test cases follow the prescribed sentence structure. FFDID, template type, record type prefix, position, and BT property or field name are all present and correctly placed.
✅
Expected results clear
Test case sentences are unambiguous. A tester can execute each case without needing to interpret intent or refer back to the SRD for clarification.
✅
No duplicates
The same test case does not appear more than once within a tab or across tabs for the same FFDID. The skill enforces this and the QA Analyst confirms it.
✅
Required Salesforce fields populated
Req ID, Test Case Title, and Test Details are fully populated in every row, the three fields required for a successful Salesforce import via the JAR tool.
Import Pipeline

From reviewed Excel
to Salesforce automatically.

Without a live MCP server connection to Salesforce, a custom JAR-based import tool was built to handle the handoff. A Power Automate flow automates the process end to end. Once the reviewed file is approved, the import runs automatically with no manual steps beyond the initial test plan setup.

Step 01
📋
Create Test Plan in Salesforce
The QA Analyst creates the test plan in Salesforce before import. This is a one-time setup per project. The test plan must exist before test cases can be associated to it.
Manual
Step 02
🔑
Add Test Plan ID to config.txt
The Test Plan ID from Salesforce is placed into config.txt. The BAT script reads this file at runtime and passes the ID to the JAR with no hardcoding required per project.
Config
Step 03
⚡
Power Automate triggers the import
After the human review checklist passes, the QA Analyst moves the reviewed Excel file into the Ready for Import folder. This triggers Power Automate to automatically execute Import_TC.bat, which calls the JAR with the Test Plan ID and the reviewed Excel file path.
Automated
Step 04
☁️
Test cases land in Salesforce
The JAR connects to Salesforce, creates test case records, associates them to the test plan, and adds them to the test run, ready for execution by the QA Analyst.
Automated
How It Works

Five steps from SRD
to test run.

The full workflow from loading the SRD to executing test cases in Salesforce test runs in five steps. The skill handles steps 1 through 3 automatically. The QA Analyst handles step 4. Power Automate and the JAR handle step 5.

01
Provide the SRD details in the prompt
The QA Analyst pastes the prompt into Claude and provides the SRD file, the mapping tab name, the FFDID, and the template type for each tab. Multiple tabs can be processed in sequence in a single run.
SRD Mapping Tab
Prompt in Claude
02
Skill processes the tab and generates test cases
The skill reads every row, identifies record type rows, applies skip logic, parses BT Properties, and generates CAPTURE and DISPLAY test cases for each eligible row according to the four case types.
The skill generates the Excel output, the execution trace log, and the skip log simultaneously. All three files use the same generation timestamp in their filename.
Confirmation Step
Completion Statement
03
Download the Excel output and review the logs
The QA Analyst downloads AI_Generated_TestCases.xlsx and reviews the execution trace and skip log to verify what was processed, what was skipped, and why, before beginning the checklist review.
Test Cases Excel
Execution Trace Log
Skip Log
04
Complete the 5-point human review checklist
The QA Analyst reviews the Excel file against the human-in-the-loop checklist, confirming coverage, format, clarity, no duplicates, and all required Salesforce fields are populated.
Any checklist item that fails stops the process. The QA Analyst corrects the input or adjusts the file, then re-evaluates the checklist before the import is allowed to proceed.
05
Power Automate imports the approved file into Salesforce
Once the checklist passes, the QA Analyst moves the reviewed Excel file into the Ready for Import folder. This triggers Power Automate to execute the BAT script, which calls the JAR with the Test Plan ID from config.txt. Test cases are created in Salesforce and added to the test run, ready for execution.
If the import is successful, Power Automate displays an Import Complete notification and moves the Excel file to the Completed folder. If the import fails, Power Automate displays an Import Failed notification and moves the Excel file to the Failed folder so it can be reviewed and reprocessed.
