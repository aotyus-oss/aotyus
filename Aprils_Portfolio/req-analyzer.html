The Problem

Requirements go into development
with unanswered questions.

Before a single test case is written, requirements need to be clear, complete, and testable. In practice they rarely are. Vague acceptance criteria, missing edge cases, conflicting rules, and assumptions buried inside plain English all make it downstream — into development, into testing, and sometimes into production.

Manual review catches some issues, but it's inconsistent, slow, and dependent on the reviewer knowing exactly what to look for. The same gaps get missed cycle after cycle.

01
Ambiguous acceptance criteria
Requirements written in plain English leave room for multiple interpretations. Developers build one thing; QA tests another.
02
Gaps in requirements
Requirements often skip null scenarios and conditional logic gaps. These surface as defects late in the cycle.
03
No systematic review
Review quality depends on who's reviewing and how much time they have. There's no consistent set of questions applied to every requirement.
The Solution

A skill that applies the same
rules to every requirement.

The Requirement Analysis Skill is installed directly in Claude. It carries a set of invisible rules, calibration tests, and output formatting instructions that load automatically every time — no setup, no configuration, no prompt engineering by the user.

The reviewer's only job is to provide the requirement text. Claude applies all rules behind the scenes and returns numbered questions targeting exactly the gaps, conflicts, and untested scenarios that manual review would miss.

How the skill works
Rules, calibration tests, and output format are baked into the skill — invisible to you and invisible to Claude's surface. Every analysis runs against the same standard, producing consistent, numbered questions in a structured Excel output regardless of who runs it or how it's invoked.

Input Modes

Two ways to run
an analysis.

The skill supports two input modes depending on whether you're doing a quick ad-hoc review of a single requirement or a structured review of an entire SRD with multiple tabs and filter configurations.

Mode 01 Ad-hoc
Paste a Requirement
Open a new Claude chat, build a short prompt using the on-page form — Req ID, FFDID (both optional), and the full requirement text — copy it to the clipboard, and paste it in. Claude runs the analysis immediately.
Best used for
Quick, ad-hoc reviews of a single requirement when you need fast feedback without uploading a file.
Mode 02 Structured
Use the Form
Configure one or more tab entries — tab name, row identification method, and filter values — then copy the generated prompt to your clipboard. Paste it into a new Claude chat, then upload or paste your SRD content.
Best used for
Structured SRD reviews with multiple requirements across multiple tabs, with precise filtering by ownership, status, or Requirement ID range.
Paste a Requirement — Step by step

Fast review in
three steps.

No file upload needed. The form builds the prompt for you — Req ID and FFDID are optional. Only the requirement text is required.

01
Open a new Claude chat
Start a fresh conversation at claude.ai. The Requirement Analysis Skill is already installed — no setup is needed. Claude will automatically load all calibration tests, rules, and output formatting the moment you send your requirement.
✅ Skill installed? Claude loads calibration tests, applies all rules, and produces the correct output. You only need to provide the text.
02
Build and paste your prompt
Use the on-page form to construct your prompt, then click Copy to Clipboard and paste it into Claude. The prompt format is:
Analyze this requirement: REQ ID: [Req ID] FFDID: [FFDID] [full requirement text]

💡 REQ ID and FFDID are optional — leave them out if not known.
💡 Use Shift+Enter to add new lines without sending.
03
Send and review your results
Claude applies all rules invisibly and returns numbered questions targeting every gap, conflict, and untested scenario found in the requirement. You only see the results — the rules are never surfaced.
Use the Form — Step by step

Structured review across
an entire SRD.

The form mode lets you configure tab-by-tab reviews with precise filtering. Each tab entry defines exactly which rows Claude should evaluate — by ownership, status, requirement ID, or both.

01
Configure your tab entries
For each SRD tab that needs review, add a tab configuration with three fields.
Tab Name — the exact name of the tab in your SRD (e.g. P4269799)
Identify rows by — choose: Ownership/Status, Requirement ID, or Both
Filter — enter the values to filter by (e.g. BT ENGINEER). Separate multiple values with commas. For Requirement ID ranges use a dash (e.g. REQ-001 - REQ-005)
02
Copy to Clipboard
Click Copy to Clipboard. Your full tab configuration — formatted as a ready-to-send Claude prompt — is copied and waiting to be pasted.
03
Open a new chat and paste
Open a new conversation at claude.ai, paste the configuration (Ctrl+V) into the chat box, and send. Claude will confirm your tab configuration and ask you to provide the SRD content.
04
Upload or paste your SRD content
Two options for providing your SRD content.
Upload the file — attach your .xlsx SRD directly in the chat
Paste the content — copy the relevant rows from your SRD and paste them in
05
Review your results
Claude applies all rules automatically across every configured tab and returns numbered questions for each eligible requirement. Output is delivered as a downloadable AI Generated Req Analysis.xlsx.
The BA Hub

Built for BAs.
Not just QA.

The Requirement Analyzer is one of two tools inside the BA Hub — a purpose-built workspace I designed for Business Analysts to accelerate analysis and requirement documentation. The hub gives BAs a single place to access AI-powered tools without needing to know how to prompt Claude.

The analyzer's primary audience is BAs reviewing their own requirements before those requirements reach development. Catching gaps, conflicts, and ambiguities at the BA stage — not at QA, not in UAT — is where this tool delivers the most value.

Why it matters at the BA stage
A requirement that reaches development with an unanswered question costs more to fix than one caught during review. The analyzer gives BAs a systematic way to close those questions before the requirement ever leaves their hands — reducing back-and-forth with developers, fewer late-cycle defects, and cleaner SRDs going into QA.
