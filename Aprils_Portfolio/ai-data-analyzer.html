Project 05 — AI · Data Analysis

AI Data
Analyzer

A Claude skill that reads an SRD mapping tab and one or more delimited data files, searches each mapped position for data, and produces a structured downloadable excel file identifying what data was found, what is missing, and where negative values exist across all files.

Type
AI Skill · Data Analysis
Input
SRD mapping tab + one or more delimited data files
Output
DATA.xlsx · SRD_to_DataFile_Mapping.xlsx · AI_Generated_Data_Trace.txt
The Problem

Missing data in the data file
is only found during testing.

Before this tool existed, there was no way to know whether the data file provided by the client actually contained values in the mapped positions. Missing data was discovered during test execution. When a test case could not be executed due to missing data, it was marked as blocked and a missing data issue raised. Testing continued on all other test cases, but blocked test case executions remained blocked until the client provided the missing data, the data was mocked, or the client approved the requirement without testing.

Core limitation
There was no way to compare the SRD mapped positions against the data file before testing began. A QA Analyst had no efficient way to scan dozens of mapped positions across multiple data files and identify which ones were missing data, or contained negative values before test execution started.
01
Missing data found too late
Data gaps were only discovered during test case execution. At that point the tester had to stop and raise an issue.
02
No cross-file visibility
When multiple data files were provided, there was no way to quickly determine which file contained data for a given position, which files were missing data, and whether any file had coverage at all.
03
Negative values in the data file were unknown before testing
There was no way to know which fields in the data file contained negative values before test execution began. The QA Analyst needed this information to test how negative data is displayed on the document.
The Solution

Know what is in the data file
before testing begins.

The AI Data Analyzer reads the SRD mapping tab and builds a mapping table linking each SRD field to its corresponding position in the data file. The QA Analyst reviews the mapping table and confirms it is correct before any extraction runs. Once confirmed, the skill searches every mapped position across all provided data files and produces a structured downloadable Excel file showing exactly what data was found, what is missing, and where negative values exist; giving the QA Analyst a complete picture before a single test case is executed.

Important distinction
The skill does not validate requirements. It confirms whether data exists at the positions the SRD maps to and identifies anything missing. It does not determine whether the data is correct or whether the mapping itself is accurate.

How It Works

Four steps from SRD
to data analysis.

The full workflow from loading the SRD to receiving the completed DATA.xlsx runs in four steps. The only pause in the process is the mapping confirmation step, where the QA Analyst reviews and confirms the SRD to data file mapping before extraction begins.

01
Provide the SRD, data files, and DOC_ID position in the prompt
The QA Analyst pastes the prompt into Claude and provides the SRD file with the mapping tab name, one or more delimited data files, and the position number that identifies the document identifier (such as invoice number or account number) in the data file.
SRD Mapping Tab
Prompt in Claude
02
Review and confirm the mapping table
The skill builds a mapping table linking each SRD field description and position to the corresponding data file column header and position. This is exported as SRD_to_DataFile_Mapping.xlsx for the QA Analyst to review and confirm before any extraction runs. The mapping is the foundation of the entire extraction. If a field is mapped to the wrong column, every result for that field will be incorrect.
This is the only pause in the process. The QA Analyst verifies that each SRD field lines up with the correct data file column before extraction runs. Once confirmed, the skill immediately proceeds without requiring any further input.
Reading Skill
SRD to Data File Mapping
03
Skill runs the full extraction
The skill scans every mapped position in every data file, identifies the first non-blank value in each, runs the negative data logic, and compiles all results into DATA.xlsx with the following tabs: All Results, one tab per data file, No Data Found, and Negative Data.
The skill also produces the full execution trace log simultaneously.
Extraction Completed
Execution Trace Log
04
Review the output before testing begins
The QA Analyst reviews DATA.xlsx to identify missing data and negative values before test execution starts. Any gaps are raised with the client to obtain data before test cases are run, preventing missing data issues during test case execution.
DATA.xlsx Output
Output Files

Multiple tabs.
Complete coverage.

The main output is DATA.xlsx, which contains multiple worksheets covering every SRD row across every data file provided. The number of tabs varies based on how many data files are provided — one tab is created per data file in addition to the fixed All Results, No Data Found, and Negative Data tabs. Each worksheet uses the same nine-column structure so results are consistent and easy to compare across tabs.

Tab 01
✅
All Results
The merged output. For each SRD row, shows the first non-blank value found across all data files in the order supplied. Every SRD row appears here, including rows where no data was found in any file.
Merged
Tab 02

One tab per data file provided. Each tab shows what that specific file contains for every SRD row, regardless of whether other files have data. Sheet name uses the base filename.

Tab 03

Data Quality
Nine-column structure
Every tab in DATA.xlsx uses the same nine columns in the same order: Requirement ID, SRD Record Type, SRD Field Description, SRD Position, Data File Header, Data File Position, Data File Value, Delimited Data File, and Document Identifier.
Key Rules

Applied the same way
every time.

The skill applies a complete, documented rule set to every SRD row and every data file. Rules cover row eligibility, mapping logic, find data logic, negative value detection, and trace output — applied in the same order with no variation.

Row Eligibility
Only mapped rows with Needs To Be Captured = Y are processed
Rows are eligible for extraction only when Needs To Be Captured = Y, regardless of the Appears On Bill value. Rows with INFORMATIONAL, N/A, or (Select One) ownership are skipped. Blank-like values are normalized before any skip logic is applied.
Mapping Confirmation
QA Analyst confirms mapping before extraction runs
The skill builds and exports the SRD to data file mapping table before any extraction begins. The QA Analyst reviews and confirms it. This is the only point in the process where the skill pauses. Once confirmed, extraction runs immediately without further prompts.
Find Data Logic
First non-blank value wins across files in supplied order
For each SRD row, the skill scans every data file in the order provided and inspects the mapped column from top to bottom. The first file containing a non-blank value is the winner. All other files are still scanned and their results recorded on their per-file tabs.
Negative Data Audit
All mapped columns scanned for negative numeric values
Every mapped column in every data file is scanned for negative values. The skill recognizes leading minus signs, parentheses notation, and currency symbols. The first negative value found per column per file is recorded. Rows with no negative values are not written to the Negative Data tab.
Document Identifier
Each result row is linked to its document identifier
The QA Analyst provides the position of the document identifier (invoice number or account number) at runtime. For every row where data is found, the value at that position in the same data file row is recorded in Column I, linking each result back to its source document.
Execution Trace
Every event logged in order
The trace log captures startup, SRD row detection, file checks, value scan results, merge winner decisions, no data found events, errors, and shutdown for every file and every SRD row.
