The Problem

Requirements written
from memory.

Writing a requirement from scratch puts the full burden on the BA to remember the correct structure, include all required fields, handle conditional logic consistently, and use the right format for the category they're writing. Miss a field and it becomes a question during QA. Write it in the wrong format and it creates confusion during implementation.

There was no standard template. No guided process. No way to ensure that a Logo requirement written by one BA looked the same as one written by another — or that the same BA wrote it the same way twice.

01
No standard format
Every BA wrote requirements differently. Same category, different structure, different fields, different wording — inconsistency across every SRD.
02
Missing fields
Requirements frequently left out FFDID, document placement, mockup paths, or conditional defaults — fields that only surface as problems downstream in QA or development.
03
Conditional logic errors
Conditional requirements — IF/ELSE logic, multi-condition scenarios, default fallbacks — were written inconsistently and often left the "what happens when none match" case undefined.
The Solution

A guided form that produces
the same output every time.

The Requirement Generator is a structured form tool that walks a BA through every field required for a given category — in the right order, with the right prompts — and produces a correctly formatted requirement ready to paste into an SRD. The BA makes every decision. The tool ensures nothing gets missed.

Important distinction
AI did not write the requirements. AI was used to build the form — the logic, the field structure, the branching, and the templates were designed and implemented with Claude as the development tool. The requirements themselves are authored entirely by the BA who fills out the form.
My Role

Designed the system.
Defined the standard.

My role was to define what a correctly written requirement looks like for every category, then design a form system that enforces that standard. That meant knowing the fields, knowing the conditional patterns, knowing what gets left out in practice — and building a tool that prevents those omissions.

I used Claude as the build tool to implement the forms, branching logic, and templates in HTML and JavaScript. Every field, every question, every template placeholder, and every logic branch reflects my knowledge of how these requirements need to be structured to survive development and QA.

Three paths per category

Three ways to work,
depending on where you are.

After selecting a category, the BA chooses how they want to work. The same three options are available for every category — the right one depends on whether the BA already has the information, is still gathering it, or wants to see the structure first.

📋 Fill out the form
The BA has all the information and is ready to build the requirement now. Step through the form, answer each field, and click Build Requirement. The tool assembles the correctly formatted requirement from the answers.
Best when the BA already has all the details from the client — FFDID, file paths, placement, conditions, and defaults.
❓ Generate questions
The BA needs to gather information from the client before writing the requirement. The tool generates a numbered question list — each question maps to a specific placeholder in the requirement template so the BA knows exactly what each answer is for.
Best when the BA is preparing for a client meeting or discovery session and needs to know what to ask.
📄 Generate a template
The BA wants to see the requirement structure with labeled placeholders they can fill in manually. The template shows exactly what the final requirement will look like — with each [PLACEHOLDER] clearly labeled so the BA can substitute the real values.
Best when the BA wants to fill in the requirement by hand or share the template structure with a client for review.
Logic Branching

Every path splits on
conditional logic.

Within each of the three paths, the BA selects whether the requirement has conditional logic or not. This single choice determines the entire form, question set, or template that follows — ensuring the correct structure for the actual requirement being written.

No conditional logic
Always the same value
The requirement applies uniformly — no conditions, no branching. A single logo, a single address, a single delivery rule that applies to every document regardless of any property value.
Display the following return address
on the document:
[RETURN_ADDRESS]
Mockup: [MOCKUP_PATH]
Has conditional logic
Value depends on a property
The requirement branches based on a BT property or Mapping field value. Different conditions produce different outputs. A default fallback handles any case not explicitly covered — and the tool enforces that the BA defines it.
IF BT Property [CONDITION_PROPERTY]
  = [CONDITION_VALUE_1],
display: [RETURN_ADDRESS_1]
ELSE IF = [CONDITION_VALUE_2],
display: [RETURN_ADDRESS_2]
ELSE: [DEFAULT_RETURN_ADDRESS]
Why this branching matters
Conditional requirements are where most requirement errors happen. The BA forgets the default case. The condition is ambiguous. The property name is wrong. By making the BA explicitly choose the logic type upfront, the tool routes them to a form that requires every branch to be defined — so nothing reaches development with a missing scenario.
How it works

Four steps from
category to requirement.

The entire flow — from opening the hub to copying a finished requirement — takes four steps. No training needed, no templates to find, no format to remember.

01
Open the BA Hub and select the Generator
Navigate to the BA Hub. Both tools are visible on the hub landing page. Click the Requirement Generator card to enter the tool.
02
Select a requirement category
Choose the category that matches what you're writing — Logo, Return Address, Remit Address, Split Email, Email/Ebill Override, or Custom Requirement. The tool loads the correct form set for that category.
Each category has its own fields, questions, and templates. Selecting the category is what determines the entire experience that follows.
03
Choose your path and logic type
Select how you want to work — Fill out the form, Generate questions, or Generate a template. Then choose whether the requirement has No conditional logic or Has conditional logic. These two choices route you to the exact right form or output.
Fill out the form — you have all the information now
Generate questions — you need to gather information first
Generate a template — you want the structure with placeholders to fill in manually
04
Build and copy your requirement
Fill in the form fields, then click Build Requirement. The tool assembles your answers into a correctly structured, fully formatted requirement. Click Copy and paste it directly into your SRD — no reformatting needed.
The output matches the exact format expected by QA and development. Field names, structure, conditional logic syntax, and placeholder labeling are all standardized across every requirement built with the tool.
